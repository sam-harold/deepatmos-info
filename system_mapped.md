# DeepAtmos System Page Map

*Real-Time Tunnel Atmospheric Hazard Detection and Alert Framework*

* Theme: Black and off White (Black accents and off white dominant backgroud), since RGY color are sensetive here.

---

## 1. Authentication & Access Control Module

### 1.1 Login Page

* Email and password authentication
* Role-based access after login

### 1.2 Password Reset

* Users must contact an administrator to reset their password
  (No automated email reset to reduce system complexity)

### 1.3 Session Management

* Automatic logout after a period of inactivity (configurable)

### 1.4 Role-Based Access Control (RBAC)

* All pages and actions are protected based on user role
* Users can only access:

  * Tunnels assigned to them
  * Functions permitted by their role

**Role Types**

* Master Admin (single account, full system control)
* Admin (manage users, tunnels, system assets)
* Technician (manage edges, nodes, operational actions)
* Associate (alert handling and operational response)
* Viewer (read-only access)

## 3. Operations Module

### 3.1 Dashboard Page

Main operational command view:

* Overall tunnel status
* Active alerts summary
* Node health summary

---

### 3.2 Live Monitoring Page

Provides near real-time operational visibility:

* Latest sensor readings
* Node heartbeat status
* Edge connectivity status
* Last communication time

**Data Flow**

* Edge devices synchronize data to the application server every **300 seconds**
* The frontend retrieves the latest data using **periodic HTTPS polling**
* Polling intervals are aligned with the synchronization cycle to reduce network overhead

When an alert is triggered, relevant sensor data is immediately available via the alert processing pipeline (Kafka event stream).

This design provides operational visibility without unnecessary real-time infrastructure complexity.

## 5. Node Management Module

### 5.1 Nodes Visualization Page

Nodes are displayed along a linear representation of the tunnel.

Color status:

* Grey: Registered but offline
* Green: Normal
* Yellow: Warning
* Red: Critical

**Layout Design**

* Each edge server covers approximately **250 meters**
* Each edge manages up to **5 nodes**
* The tunnel is divided into **1 km segments**
* Each segment may contain up to:

  * 4 edge servers
  * 20 nodes

Nodes are clickable and open the Node Detail page.

---

### 5.2 Node Detail Page

Displays:

* Historical sensor graphs
* Last seen timestamp
* Associated edge device
* List of alerts generated by the node

**Actions**

* Enable / Disable node

Disabled nodes:

* Continue logging data
* Do **not trigger alerts**

# Architectural Communication Model (Summary)

DeepAtmos uses a hybrid communication model:

* **Batch Telemetry**
  Edge → Cloud every 300 seconds (sensor readings)

* **Event-Driven Processing**
  Alerts published via Kafka for immediate processing

* **Frontend Data Access**
  Periodic HTTPS polling aligned with batch interval

* **Mobile Notification**
  Firebase Cloud Messaging (FCM) for critical alerts

This approach balances responsiveness, system stability, and operational simplicity.
